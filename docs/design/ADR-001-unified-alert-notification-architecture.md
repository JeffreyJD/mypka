## Why this ADR exists, and a note on scope

This started as five parallel domain surveys (email, home automation, homelab, network, VPS/software) plus external research, aimed at one incident: YoLink fridge/freezer alarms fired almost daily for a month, buried in email, unnoticed. Partway through drafting, Jeff reframed the priority directly: **the goal is not a one-time inbox cleanup — it's a standing, always-on assistant with a continuous view of everything landing in his inboxes (plural), not a tool he has to remember to invoke.** Then, further into drafting, Jeff ruled directly on *how* that standing assistant executes for v1: **run it through the team itself** (Claude Code, the existing specialist roster) rather than externalizing it to separate automation infrastructure — explicitly framed as a v1 sequencing call, not a permanent rejection of the externalized model. Both reframes change what ships first and how, not the underlying five-domain design. This ADR reflects all of it: it bundles two coupled decisions Jeff should approve together, because the second depends on the first's output.

- **Decision 1 (primary, near-term critical path):** how the email pipeline (GL-018/SOP-023, already designed and working in dry-run) becomes a *standing* multi-inbox assistant instead of an on-demand tool — and, per Jeff's latest ruling, how that standing assistant runs **within the team's own Claude Code orchestration for v1**, not on separate automation infrastructure.
- **Decision 2 (complementary, ships in parallel where unblocked, later where not):** the shared cross-domain relay + severity scheme that home automation, homelab, network, and VPS/software alerts all funnel into — with email's `ALERT`-class output as one of its inputs, not a separate system.

Both share two non-negotiable cross-cutting rules: **human-in-the-loop confirmation**, generalized here from GL-018's email-specific rule into a system-wide invariant, and **no historical backfill on onboarding** (see "Cross-cutting decision" below for both).

A note on why this is one ADR covering two decisions rather than two: they were commissioned together, they share the confirmation/severity framework, and Decision 2's only live input today is Decision 1's output (email `ALERT` items). If a future change needs to revise only one of them, split it into a superseding ADR at that point rather than editing this one in place — see Keystone's standing rule against silent edits. (This document has gone through several rounds of live revision during drafting, per Jeff's own corrections and parallel specialist research arriving mid-session; none of it is "accepted" yet, so revising in place here is the correct move, not a violation of that rule — the rule protects an *approved* ADR from silent edits, not an in-progress draft.) **This is intended as the final drafting pass** — Decision 1 and Decision 2 are now fully decided for v1, with only a small, named set of genuine unknowns still open (see the closing section).

## Decision 1 — The email pipeline becomes a standing, scheduled, multi-inbox assistant, running within the team for v1

### Context

GL-018 and SOP-023 are fully designed and already shipped (taxonomy, three-tier priority override, propose-then-confirm, PKM routing). But as written, SOP-023 states plainly: it "runs on-demand only... or on a schedule once a trigger layer exists" — **that layer does not exist.** Today "the assistant" is a manual invocation inside an LLM session. That is structurally the same failure class as the incident that started this work: a system that only helps when a human remembers to ask it to. Only one inbox (`gmail-personal`) has actually been dry-run; `davisglobe` is architected for (`inbox_id`-scoped state already exists) but not onboarded. (`perficient` is out of scope entirely — an earlier draft carried it over from a stray session-log note that was itself wrong; Jeff confirmed it has nothing to do with this effort, and GL-018 has already been corrected to remove that reference.)

**Four addresses, two tracks (new context, from a parallel research pass).** B.J.'s follow-up research ([[Deliverables/2026-07-21-gmail-contacts-automation-architecture-research|gmail-contacts-automation-architecture-research]]) established that "sending replies on approval" actually spans four addresses splitting into two tracks with genuinely different mechanisms:

- **Track A** — `jeffreyj2490@gmail.com` and `davisglobe@gmail.com` (durably Google, no migration risk). Recommended mechanism: a self-hosted, Vex-reviewed community MCP server (`taylorwilsdon/google_workspace_mcp`) wired into Radar's shim, giving it full Gmail read/draft/send + Contacts CRUD tools — the same self-owned-GCP-OAuth-client foundation Klinger already scoped, just routed through a mature server instead of hand-rolled code. **This ADR's execution-model pivot does not change Track A** — the MCP server is the right way to give Radar's shim these tools regardless of which mechanism invokes it.
- **Track B** — `jeff@davisglobe.com` and `admin@davisglobe.com` (confirmed migrating off Google Workspace to an undecided destination). Originally scoped to n8n + IMAP/SMTP. **Resolved:** Jeff ruled directly — n8n is not used for v1. Track B runs as its own Desktop Scheduled Task on `jeff-laptop`, invoking Klinger's short, auditable stdlib Python script (`imaplib`/`smtplib`, standard library only, no third-party package) for fetch/draft/send. See the dedicated subsection below for the full reasoning.
- **A scope-name correction, applying everywhere in this ADR:** every earlier reference to a `gmail.send`-scoped connector was wrong. The actual operation needed for drafting and sending (`drafts.create`/`drafts.send`) requires the `gmail.compose` scope (B.J.'s Finding 2) — `gmail.send` alone doesn't cover it. Every mention below is corrected to `gmail.compose`.
- **A gap not yet closed:** GL-018/SOP-023 currently name only two `inbox_id`s (`gmail-personal`, live; `davisglobe`, architected but not onboarded). Four addresses across two tracks likely need more than two `inbox_id`s once fully onboarded (e.g., is `davisglobe` today's placeholder for `davisglobe@gmail.com`, `jeff@davisglobe.com`, or both?). This ADR does not resolve that mapping — flagged as open below, not assumed 1:1. This is the one genuine unknown left in the address/track picture; everything else about Track A and Track B is now decided.
- **A side benefit of the execution-model decision, worth naming here:** since Track A and Track B both run as Desktop Scheduled Tasks on the same machine (`jeff-laptop`), the Contacts-cache hand-off between them (Track A produces a cached name+email snapshot; Track B reads it for any contact-matching need it turns out to have) is just a local file read — no cross-host bridge, no network hop, no separate credential to secure that hop with.

Separately, Jeff confirmed sending email replies on his approval is a "must have" — resolved to mean **both** draft and immediate-send, selectable per message, not a single fixed pipeline choice (see "Resolved — draft vs. send").

**The execution-model pivot this revision addresses.** Partway through drafting, Jeff ruled directly: *"for initial launch let's run through the team. down the road we have to change, but right now let's start with architecting this within the 4077th."* This is an explicit v1 sequencing decision, not a rejection of externalized automation infrastructure — the original plan (a VPS-hosted script calling the Anthropic API directly, and/or n8n for Track B) is retained as the named, deliberate **v2+ migration path**, not abandoned. Everything in this Decision below reflects that ruling.

**A mechanism correction, now fully resolved, not open.** An earlier pass of this section assumed the mechanism was a `/schedule` **Cloud Routine**. Klinger researched this directly (confirmed against Claude Code's own documentation): **Cloud Routines run in Anthropic's cloud against a fresh clone of the GitHub repo and can only write back via PR branches — they do not have live, direct vault access, full stop.** This is a structural mismatch with SOP-023, which needs live, direct, same-day reads/writes to `PKM/Journal/`, `PKM/CRM/`, the sender registry, and `.mypka/inbox-state/*.json`. Worse, a Cloud Routine would require Track A's OAuth refresh tokens to live as cloud-sandbox secrets, a real downgrade from this vault's "nothing uploaded" posture. **The correct v1 mechanism is Desktop Scheduled Tasks** — a Claude Code Desktop app feature (distinct from the CLI) that runs a real local session on `jeff-laptop`, against the actual live vault, using the same local `.mcp.json` any interactive session would use, with credentials never leaving the machine. Everything below uses Desktop Scheduled Tasks, not Cloud Routines. (Put plainly: Cloud Routines' clone/branch/PR model is actually the *more* externalized of the two mechanisms, not the less — Desktop Scheduled Tasks is the more faithful "within the 4077th" implementation of Jeff's ruling.)

### The real question

Not just "where does SOP-023 run when nobody is in a chat session" but **what kind of thing runs it** — a piece of separate automation infrastructure, or the team's own agents, on a schedule, doing exactly what they already do interactively, against the real vault? Jeff has answered this for v1: **the team's own agents, via a real local session on `jeff-laptop`.** Concretely: a Desktop Scheduled Task invokes Radar, who executes SOP-023 directly — same procedure, same taxonomy, same specialist identity that already runs this interactively, in the same working directory an interactive session would use — using whatever MCP tools (or, for Track B, plain scripts) are wired into Radar's local `.mcp.json` / tool access (today: four read-only Gmail tools; once built: Track A's `google_workspace_mcp` server for the two durably-Google addresses, and Track B's stdlib script for the two migrating addresses).

myPKA still has no runtime of its own outside the team's own orchestration — "the folder is the system," no build step, no code execution inside `PKM/`, `Team Knowledge/`, etc. What changes under this pivot is *not* that constraint; it's that "unattended execution" now means "the team, on a timer, on the same machine and vault a human session would use" rather than "a separate script calling an LLM API from outside the team's own identity and tooling." The VPS's proven cron-calls-Anthropic-API pattern (`weekly_strategy_report.py`) remains a valid, working precedent for *externalized* automation — it just becomes the **v2+** answer rather than the v1 one, per Jeff's explicit sequencing call.

### Options

**Option 1 — v1 (Jeff's explicit ruling): a Desktop Scheduled Task invokes Radar to run SOP-023 directly, on `jeff-laptop`.**
A recurring Desktop Scheduled Task, configured in the Claude Code Desktop app, wakes on a cadence and runs a real local session — Radar through SOP-023, for each onboarded inbox: fetch (via whatever MCP tools/scripts are wired in), classify (GL-018's taxonomy, loaded the same way it already is interactively), execute `TRASH`/`ARCHIVE` where GL-018 permits it without a nod, push every `ALERT`/`ACT` proposal through the shared relay (Decision 2) immediately, and — because this is a real local session against the live vault, not a cloned/sandboxed one — execute `FILE` writes directly, the same way an interactive session already can (GL-018's `FILE` disposition was already permitted to auto-execute without a per-item nod even interactively; this isn't a new policy, just the same permission exercised on a timer).
- *Cost:* Low — no new VPS, no new host-level credential, no new deploy target. The mechanism is a Claude Code Desktop app feature, not infrastructure Jeff provisions.
- *Risk:* Genuinely low blast-radius — no new external credential is minted at all; the scheduled task uses the same identity (Radar), the same local `.mcp.json`/scripts, and the same vault-write permissions the team already trusts interactively. Credentials (including Track A's OAuth refresh tokens and Track B's IMAP/SMTP credentials) never leave `jeff-laptop`.
- *Reversibility:* High — disabling a Desktop Scheduled Task is a configuration change, not a credential-revocation or host-teardown event.
- **Prerequisite, resolved and in progress:** Desktop Scheduled Tasks require the Claude Code **Desktop app** specifically, distinct from the CLI Jeff runs day-to-day (`PKM/Environment/Hosts/jeff-laptop.md` lists "Claude Code" without previously distinguishing CLI vs. Desktop app). **Jeff is having Bastion install the Desktop app on `jeff-laptop` now**, additive alongside his existing CLI usage — the CLI stays Jeff's primary interactive tool; the Desktop app's only job is enabling this feature. First item in the sequencing table below, status: in progress.
- **One real reliability cost, stated plainly, not hidden:** a Desktop Scheduled Task only fires while `jeff-laptop` is awake and the Desktop app is running — unlike a VPS, which is always on. The mechanism catches up on one missed run per wake (if the laptop was asleep when a scheduled fire was due, it runs once on next wake); anything older than that one catch-up window is dropped, not queued indefinitely. This is a genuine v1 limitation — a laptop that's closed all weekend means no triage runs all weekend — consistent with Jeff's own "down the road we have to change" framing, not a defect this ADR should paper over. Jeff has confirmed this tradeoff is acceptable for v1 (see Decision 2's heartbeat section, where the same tradeoff applies to the canary and has been explicitly accepted rather than engineered around).
- *This is Jeff's explicit ruling for v1, not a Keystone recommendation weighed competitively against Option 2 this round* — though it happens to also come out ahead on cost and credential-security posture versus the original plan.

**Option 2 — v2+ (deferred, explicitly named, not rejected): externalize to automation infrastructure outside the team's own orchestration.**
The original plan: a scheduled script/service on `davisglobe-vps-ash-1` (or a dedicated host) calling the Anthropic API directly, external to any Claude Code session — plus, for Track B specifically, n8n's protocol-generic IMAP/SMTP/CardDAV nodes (no longer the v1 plan for Track B, but still a legitimate v2+ option if Klinger's stdlib script ever needs replacing). This remains the "true always-on, laptop-independent" answer — it doesn't share Option 1's laptop-awake reliability limit — and is explicitly retained as the direction to revisit **once the v1 model's real limits are known**, e.g., how much the laptop-awake gap actually costs in practice.
- *Cost:* Medium — new host or hosted process, new credential class (Drive API, if full unattended vault writes are wanted on a VPS instead of a laptop; n8n's own IMAP/SMTP credentials for Track B, if n8n is ever revisited), new deploy target under WS-006.
- *Risk:* This was the original ADR's largest-blast-radius option, and that analysis still applies if/when it's built — see the retained reasoning below. Notably, this is now understood to be a *lower*-credential-risk profile than a Cloud Routine would have been (Option 2 keeps credentials on a VPS Jeff controls directly, rather than an Anthropic-cloud sandbox), but still a real new credential surface relative to Option 1's laptop-local model.
- Named explicitly per Jeff's own "down the road we have to change" framing — a planned future migration, not a strawman.

**Option 3 — Do nothing (keep SOP-023 on-demand only).**
Stated because it is the honest baseline, not a strawman: zero new cost, zero new risk. Rejected because it is the exact status quo that produced the incident this whole effort exists to fix, and it directly contradicts Jeff's explicit reframe that on-demand is not the goal.

### Recommendation

**Option 1, per Jeff's direct ruling, implemented via Desktop Scheduled Tasks specifically (not Cloud Routines).** Option 2 is retained as the named v2+ direction — not evaluated against Option 1 on cost/risk this round, since Jeff has already decided the sequencing; it stays in this ADR so the reasoning (and the credential/security-review weight it carries) isn't lost when the team does eventually revisit it, most likely triggered by Option 1's laptop-awake reliability limit becoming a real, felt cost rather than a theoretical one.

### Vault write-access scope — resolved

Cloud Routines (the `/schedule` mechanism) do **not** have live vault access — they operate against a fresh clone of the GitHub repo and can only write back via PR branches. Desktop Scheduled Tasks **do** have live vault access — they run a real local Claude Code session on `jeff-laptop`, in the actual vault directory, the same way an interactive session does. `FILE`/`ARCHIVE`/`TRASH` execute directly and immediately under the mechanism this ADR specifies, with no staging or PR-based reconciliation needed. Source: Klinger's research pass, confirmed against Claude Code's own documentation.

### Where does the standing runner live?

**`jeff-laptop`**, via a Desktop Scheduled Task — not a VPS. The original ADR's "co-locate on `davisglobe-vps-ash-1` (Option A) vs. a new dedicated VPS (Option B)" framing assumed a VPS-hosted process and doesn't apply under Option 1 at all; there is no host-provisioning decision to make for v1. That framing is retained below for whenever Option 2 (externalized automation infrastructure) is actually built at v2+, at which point it becomes live again.

*(Retained for v2+ reference: `davisglobe-vps-ash-1` is tagged `trading` end-to-end and runs Prophet Trader's real trades — Option A there was "co-locate, lowest cost, blends two blast radii onto one host"; Option B was "a new, small, dedicated VPS for personal-automation workloads, ~$5-10/mo, clean separation." Revisit this exact framing if/when v2 externalizes the runner.)*

### Multi-inbox onboarding

Per Jeff's reframe, onboarding beyond `gmail-personal` is near-term, not backlog — GL-018/SOP-023 are already `inbox_id`-scoped for exactly this. (`perficient` is explicitly out of scope — see the Context note above.) **The exact set of `inbox_id`s to onboard should be reconciled against the four-address, two-track breakdown above before proceeding** — today's ADR draft still names only `davisglobe` as the next inbox, which may need to become two or more distinct `inbox_id`s (one per address, or one per track) once that mapping is settled; treat "onboard `davisglobe`" below as a placeholder for "onboard whichever `inbox_id`(s) the resolved address mapping produces." Sequencing otherwise unchanged: each new inbox starts in `dry-run` per SOP-023's existing ramp discipline (~2 weeks of Jeff watching classifier precision before graduating to live), same as `gmail-personal`. This is Radar's existing procedure, unchanged — the only new fact is that once the Desktop Scheduled Task exists, "dry-run" for a newly onboarded inbox means "push alerts, stage everything else," not "print to a chat session nobody's watching."

**Design lock — no historical backfill on onboarding.** When a new inbox goes live, its per-inbox state marker starts at the onboarding timestamp — the standing runner processes only mail arriving from that point forward. There is no historical sweep as part of standard onboarding. The `gmail-personal` inbox's original 30-day backfill (the live cleanup that seeded GL-018 itself) was a one-time, separately-approved exception to bootstrap that specific inbox's cleanup at the start of this whole effort — it is not the onboarding pattern going forward and must not be repeated by default. If Jeff ever wants a genuine historical sweep of a newly onboarded inbox again, that is a fresh, explicit, separately-approved ask each time — never something the onboarding procedure does on its own initiative.

### Resolved — draft vs. send: both, selected per-message at approval time

**Jeff has ruled on this; it is no longer an open question.** He wants both paths available simultaneously, not a single fixed pipeline choice made once at design time: at the moment he approves a drafted reply, he chooses, per message, either **"approve as draft"** (creates the draft only; Jeff sends manually) or **"approve and send"** (fires immediately, no manual step). This is a per-message decision at approval time, not a global setting or a one-time architecture pick between the two.

**Track A and Track B each have their own mechanism for this, both now settled.** Track A's two addresses (`jeffreyj2490@gmail.com`, `davisglobe@gmail.com`) route both outcomes through the `gmail.compose` scope (corrected per B.J.'s Finding 2, not `gmail.send`) on the `google_workspace_mcp` server. Track B's two addresses (`jeff@davisglobe.com`, `admin@davisglobe.com`) route through Klinger's stdlib script: `approve_as_draft` → an IMAP append to a Drafts folder via `imaplib`; `approve_and_send` → an SMTP send via `smtplib`. Same per-message choice, same two outcomes, different mechanism underneath per track.

**Architectural consequence:** the `send_reply(message)` interface seam needs to carry both the approval outcome and enough context to route to the right track — effectively `send_reply(message, outcome, track)` where `outcome ∈ {approve_as_draft, approve_and_send}` and `track` resolves which underlying mechanism actually executes it:
- `approve_as_draft` (Track A) → calls the MCP server's draft-create tool (`gmail.compose` scope).
- `approve_and_send` (Track A) → calls the MCP server's send tool (same `gmail.compose` scope covers both).
- `approve_as_draft` (Track B) → Klinger's script appends to an IMAP Drafts folder.
- `approve_and_send` (Track B) → Klinger's script sends via SMTP.

**This connector is a required build, not an optional path** — "approve and send" cannot function at all without Track A's `google_workspace_mcp` server (gated on Vex's review and Jeff's Cloud Console session) and Track B's stdlib script (gated on a lighter Vex read-through, see the Track B subsection below) both being wired in.

**Note for whoever implements the approval surface** (wherever that lives — chat, Telegram, or otherwise): every drafted-reply proposal must present **both** options clearly per message — "approve as draft" and "approve and send" — never a single binary "approve?" that only exercises one path. This is a UI/prompt-design requirement on top of the interface seam, not something the interface alone guarantees.

### Track B's execution mechanism — resolved: Klinger's stdlib script, via its own Desktop Scheduled Task

**Jeff has ruled on this directly** — "that script is fine to start with." **Track B uses Klinger's short, auditable stdlib Python script** (`imaplib`/`smtplib`, standard library only — no third-party IMAP/SMTP package, no n8n), invoked by its own Desktop Scheduled Task on `jeff-laptop` — the same mechanism as Track A, just calling a plain script instead of an MCP server.

This resolves the reasoning an earlier draft of this ADR laid out as an open question (kept here for the record, since it's still useful context for *why* this is the right v1 answer, not just *what* was decided):
- Consistency with Jeff's "run within the team" principle — Track B now runs on the same machine, same scheduling mechanism, same credential-locality posture as Track A, rather than standing up n8n as a whole separate platform (Postgres, a persistent container, ~2 vCPU/4GB RAM) for one of the two tracks only.
- A stdlib script sidesteps, rather than answers, the "does a mature IMAP/SMTP MCP server exist" question this ADR previously left open for B.J. to research — a short, auditable, standard-library script is a lower-trust-surface answer than either a third-party MCP package or a whole n8n platform would have been, and needed no further research to justify for v1.
- IMAP/SMTP remains the protocol-portable choice B.J.'s original brief identified (Finding 6) — that portability argument is about the *protocol*, not the execution engine, and holds identically whether the protocol is driven by n8n, a third-party MCP server, or Klinger's own script.

**Revisit only if a more mature, independently-reviewed MCP option for IMAP/SMTP emerges later** — this is a deliberate v1 answer, not a permanent ban on ever adopting one.

**One gate this still needs, distinct from Track A's:** Klinger's script is standard-library-only, so it doesn't carry the third-party-package provenance risk `google_workspace_mcp` does — but it still touches live mail credentials and sends real outbound messages, which is enough to warrant a quick Vex read-through of the script itself (a correctness/security skim, not a full package-trust audit) before credentials are wired to it, per WS-006's "security review if it touches auth, data, secrets, or an endpoint" standard. Lighter than Track A's review, but not zero.

## Decision 2 — The shared cross-domain relay + severity scheme

### Context (five domains, current state)

| Domain | Owner | Current state |
|---|---|---|
| Email | Radar | Fully designed (GL-018/SOP-023); becomes standing per Decision 1 |
| Home automation (YoLink/Ring/Nanit) | Relay | Zero Environment footprint; incident-causing device (YoLink) confirmed unmonitored for a month |
| Homelab (Lighthouse/Watchtower/OPNsense) | Trapper | Not built; 0%-mid-assembly. Nothing to monitor yet. |
| Network (UniFi) | Sparky | Live, dashboard-only, zero push. Confirmed 5-week blind spot (unadopted switch). |
| VPS/software | Pierce | Most mature: Healthchecks.io (cron dead-man's-switch, working) + a Telegram bot (hardcoded, `prophet-trader`-only). GitHub Actions failures land as unread platform email — two recent silent misses. |

B.J.'s research (full detail in the brief) converges on: Healthchecks.io and Uptime Kuma solve different failure classes and should coexist; generalizing the existing Telegram bot is the lowest-friction fix available today; Home Assistant, once real, should be the single home-domain upstream; a 3-tier severity scheme (critical/needs-attention/informational) fits solo-operator scale; and — the single most load-bearing finding — **the relay itself needs an external heartbeat, because "silently stops alerting" is structurally the same failure this whole effort exists to fix, one level up.**

Note: these five rows describe the five *domains of origin* — who owns wiring each source into the relay. They are not the same as the four *delivery channels* Jeff wants (see "Channel structure"); several domains share one channel by design.

**Scope note on the execution-model pivot:** Decision 1's "run within the team for v1" ruling is specific to judgment-heavy knowledge work that Radar (and, by extension, the team) already does interactively — email triage. It does **not** imply that prophet-trader's own trading code, homelab builds, or network configuration should also move "into the team's" execution model — those are Pierce/Trapper/Sparky's existing software/infrastructure lifecycles (WS-006/WS-007), unrelated to whether an LLM agent is doing the judgment work. Decision 2's relay and per-domain ingestion are otherwise unaffected by the pivot, with two exceptions (the heartbeat-canary, and the relay's callable shape), both resolved below.

### The shared relay's shape — where does it live and who calls it?

Today the Telegram bot is "a plain Bot API POST call... trivial to generalize" but embedded inside `prophet-trader`'s own repo and `.env` — nothing outside that repo can reach it. This sub-question is about *where the relay's code and ownership live* — a separate question from *what transport/channel structure it uses*, addressed next.

**Option 1 — Generalize in place: expose an HTTP endpoint or importable module from within `prophet-trader`.** Lowest effort. Rejected as the end-state: it makes an unrelated project (trading) the de facto host for infrastructure that isn't about trading, coupling two deploy lifecycles and gate tiers that should stay independent (prophet-trader is Full-tier, cost-of-failure-gated; a notification relay is not a trading decision and shouldn't inherit Blake's phase-gate ceremony by association).

**Option 2 — Extract to a small, standalone, source-and-severity-tagged relay** (a callable script/CLI + thin importable interface — not necessarily a persistent daemon), owned outside `prophet-trader`. Every producer calls one interface: `send(source, severity, message, metadata)`. **Under Decision 1's model, "every producer" spans two different execution substrates**: Claude Code sessions on `jeff-laptop` (Decision 1's Desktop Scheduled Tasks, invoking the relay via a Bash-tool call to the shared script) and plain, non-Claude-Code processes (prophet-trader's own Python code on the VPS, GitHub Actions webhooks) — the relay's interface needs to be callable from both, which just means it stays a small, dependency-light script/module rather than something that assumes a specific caller runtime. This doesn't change the recommendation, only clarifies it.

**Recommendation: Option 2**, invocable from either a Claude Code session's tool access or a plain script/process, unchanged in substance from the original recommendation.

### Channel structure — real separation by group, not tags-in-one-stream

**This is a revision to the transport recommendation below, not to the interface or the location decision above.** Jeff has stated a requirement stronger than "one critical channel vs. one general channel": distinct channels per group of alerts, so he can mute or check each group independently — real separation, not message-text tags inside a single stream.

An earlier working interpretation of the grouping (five channels, one per domain of origin) has been **directly corrected by Jeff** to the following **four channels**, which supersedes that assumption and is now decided, not inferred:

1. **Trading** — Prophet Trader specifically (fills, crash notifications, EOD summaries, anything about the trading process itself). Kept fully separate from general infrastructure even though both run on the same VPS — a trading alert and a disk-space warning are not the same kind of thing to Jeff, regardless of which box produced them. `source: prophet-trader`.
2. **Infrastructure** — one combined channel for everything technical/ops: VPS non-trading concerns (`source: vps` — disk/memory/systemd, GitHub Actions CI/deploy failures across repos), homelab once built (`source: homelab` — Lighthouse/Watchtower/OPNsense), and network/UniFi (`source: network`) — **plus backups specifically**, called out by Jeff by name (`source: backup` — VPS rclone-to-B2 backups today; future Proxmox backups and OPNsense config backups once homelab exists). All of this is one group, not three or four separate ones.
3. **Home automation** — YoLink/Ring/Nanit/Home Assistant (`source: yolink`, `source: ring`, `source: nanit`, `source: home-assistant`), its own channel.
4. **Email** — unchanged from the original ask (`source: email`); already its own channel via GL-018/SOP-023's existing tiering, not folded into Infrastructure.

Severity (critical vs. informational) remains a **property within each of these four channels**, exactly as before — this correction changes only the grouping axis (four channels, grouped as above, instead of five-by-domain-of-origin), not the severity/confirmation logic, which is unaffected.

**One inference this ADR makes, flagged for confirmation rather than asserted as settled:** Healthchecks.io's three trading-cadence checks (Daily Routine, Weekly Autopsy, Weekly Strategy Report, Daily Fidelity Check) monitor whether the *trading process itself* ran — this ADR reads them as **Trading**-channel content, alongside fills/crash notifications, rather than Infrastructure. GitHub Actions CI/deploy failures — even for the `prophet-trader` repo itself — are read as **Infrastructure**-channel content per Jeff's explicit "VPS non-trading... GitHub Actions failures" example, since a deploy/CI failure is an ops concern distinct from whether the strategy is trading. This Trading-vs-Infrastructure split for Healthchecks specifically is Keystone's inference from Jeff's "non-trading" qualifier, not something Jeff stated as a rule for Healthchecks by name — worth a quick confirmation with Jeff when Pierce wires it, cheap to flip if wrong. **This remains the one live open question in Decision 2's design**, alongside the exhaustive-membership details noted below.

This still changes what "the relay" must support. The transport reassessment holds, updated to four channels instead of five:

- **Multiple Telegram bot tokens, one per channel (four total).** Each channel is a distinct bot + chat, called through the same relay interface (`send()` resolves `source` to the right channel, and each channel to its token/chat_id, before posting). The existing `prophet-trader` bot is reused directly as the **Trading** channel's bot — an exact fit, since that bot's entire original purpose (fills, crash alerts, EOD summaries) *is* the Trading channel's content. Three new bot tokens are needed: **Infrastructure**, **Home Automation**, **Email**. Real separation, mutable/checkable independently in the Telegram app Jeff already uses daily. Bonus: this also *reduces* the single-point-of-failure blast radius B.J.'s Finding 6 warns about — a revoked token today would silently kill 100% of alerting; under this model it silently kills one of four channels, which is itself more likely to be *noticed* precisely because the other three keep working normally. Cost: three new bot registrations via BotFather (trivial, same mechanism as the existing bot), three additional credentials to store per [[Team Knowledge/Guidelines/GL-013-credential-backup-hygiene-on-operational-hosts]] — a modest, not a large, increase in secret-management surface.
- **A single Telegram supergroup with Topics enabled** (one topic per channel). Real separation from Jeff's point of view (mutable per-topic), but it does **not** reduce the SPOF risk above — one bot token still underlies every topic, so a single revocation still blacks out all four channels simultaneously, same blast radius as today. It also requires standing up and administering a supergroup, more ceremony than registering three more bot tokens for no corresponding benefit here. **Rejected** relative to the multiple-bot-tokens option above — strictly worse on the one axis (blast radius) that matters most per B.J.'s own findings, for no offsetting gain.
- **ntfy**, per B.J.'s research, has first-class topics with per-topic access control — the most structurally "correct" fit for named, mutable channels, and each topic can carry its own access token, which would *also* reduce the SPOF blast radius (a leaked/revoked topic credential affects one topic, not all). The cost is real and was already characterized in B.J.'s brief before this requirement sharpened it: a new self-hosted service (or ntfy.sh cloud account) to deploy and keep healthy — itself a new moving part needing its own place in the heartbeat story — plus a new mobile app for Jeff to install and learn, versus Telegram, which he already uses daily for exactly this purpose.

**Revised recommendation: multiple Telegram bot tokens (one per channel, four total) for v1**, reusing the existing `prophet-trader` bot as the Trading channel. This satisfies the real-separation requirement immediately, at the lowest incremental cost, with zero new app or service for Jeff to adopt, and it is a strict blast-radius improvement over today's single-bot setup rather than a lateral move. **ntfy remains the pre-planned upgrade path** — per B.J.'s original guidance ("worth it once volume grows past a few/week"), now sharpened by this requirement to also read "worth it once Jeff wants per-topic credential scoping, not just per-topic display separation" — and the `send(source, severity, message, metadata)` interface is deliberately built so that swap changes only the destination-resolution logic inside the relay, not any producer's call site. This is the same abstraction benefit already designed into Decision 2 before this revision; it holds without rework.

**Nothing left open on the grouping axis itself** — Jeff has directly specified the four channels above. What remains open is only the *exhaustive membership* of each channel's sub-sources as new sources come online (e.g., confirming Proxmox/OPNsense backup-alert specifics once homelab exists, confirming Home Assistant's exact event types once real, and the Healthchecks Trading-vs-Infrastructure inference flagged above) — these are detail questions for the owning specialist (Trapper/Relay/Sparky/Pierce) to resolve when each source is actually wired, not open architecture questions.

### Severity tiers vs. email priority tiers — reconcile or separate?

**These are two different axes and forcing them into one scheme is a category error.** GL-018's priority-tier override (Tier 1 contact match / Tier 2 direct-human-correspondence / Tier 3 category-fallthrough) answers a **who** question — does the *sender* deserve to jump the queue? It presupposes a sender, which most non-email alerts don't have (a fridge alarm has no "contact"). B.J.'s severity scheme (Tier 1 critical / Tier 2 needs-attention / Tier 3 informational) answers a **what** question — how dangerous or costly is *this event*, independent of who or what produced it.

**Decision: keep them explicitly separate, named distinctly (GL-018 keeps "priority tier," the cross-domain scheme is "severity tier"), and connect them with one mapping rule rather than merging them:**

| Email disposition (GL-018) | Cross-domain severity (this ADR) | Rationale |
|---|---|---|
| `ALERT` (any priority tier, any category) | **Severity 1 — push now** | GL-018's `ALERT` already means "surface to Jeff now"; that's definitionally Tier-1-severity the moment it's pushed through the relay instead of left in an in-run digest. |
| `ACT` (no urgency signal — a routine task/reply proposal) | **Severity 2 — needs attention, not urgent** | Held for confirmation regardless; muted/digest delivery is enough. |
| `ACT` where the Tier-1/2 priority override also fired | **Severity 1** | A contact-match or direct-human message needing action doesn't wait for a digest. |
| `FILE` / `ARCHIVE` / `TRASH` | **Not in the alerting layer at all** | These never surface; no severity applies. |

This is the whole reconciliation: one small mapping table, not a unified taxonomy. Any future domain (home, homelab, network, VPS) just assigns severity directly per B.J.'s 3-tier scheme — it never touches the email priority-tier concept, because it has no sender to evaluate. **Per the channel-structure revision above, severity determines how a message behaves *within* its channel (loud vs. silent vs. logged-only) — it does not create a second set of channels alongside the four channels defined above.**

### Per-domain ingestion (only where a real decision exists today)

- **Email → relay:** Decision 1's standing runner posts directly, tagged `source: email`, landing in the **Email** channel; wiring is push, not pull.
- **VPS/software → relay:** splits across two channels, not one. Pierce generalizes the existing Telegram send call (Option 2 above) for prophet-trader's own trading alerts (fills, crash, EOD, and — per this ADR's inference above — the Healthchecks.io trading-cadence checks) → **Trading** channel, `source: prophet-trader`. Separately, Pierce adds a GitHub Actions failure-notification step that curls the relay on `workflow_run: failure` (any repo, including `prophet-trader`'s own CI) and wires general VPS host health (disk/memory/systemd) and the rclone backup job → **Infrastructure** channel, `source: vps` / `source: backup`. This directly closes both known silent misses (prophet-trader deploy, mypka snapshot-notify). This piece is entirely unaffected by Decision 1's execution-model pivot — it's plain Python/cron code, not a Claude Code session.
- **Home automation → relay:** Home Assistant is the correct long-term single upstream for this domain once it's real (YoLink and Ring both have documented HA integration paths). HA is not real today — it's an onboarding-wizard sandbox on `bridget-laptop`, stood up for an unrelated project. **This ADR does not design HA's eventual architecture** — that's premature ceremony against hardware/software that doesn't functionally exist yet for this purpose. What it does require now: Relay establishes *some* interim direct-to-relay path for the one device class that has already caused real harm (YoLink temp/offline alarms) — mechanism (webhook vs. polling script) is Relay's technical call, not this ADR's; the requirement is only "tag `source: yolink`, assign severity per event type, post to the relay" (landing in the **Home Automation** channel) without waiting for full HA to exist.
- **Network → relay:** Sparky needs a live UniFi controller login to confirm firmware version (governs native webhook support). If present, wire directly; if not, a lightweight polling script bridges the gap until Kuma exists. Either way, closing the 5-week blind spot (adoption-status/device-count check) doesn't need to wait for that firmware check — a simple scheduled probe pushed to the relay (`source: network`, landing in the **Infrastructure** channel, not a separate network channel) can ship regardless of what the firmware check finds.
- **Homelab → relay:** Nothing to build. Lighthouse/Watchtower/OPNsense don't exist yet. The only decision worth recording now is the *principle* — once built, Uptime Kuma is the natural homelab+network probe layer (per B.J.), feeding the same relay (`source: homelab`, plus `source: backup` for Proxmox/OPNsense config backups specifically) into the **Infrastructure** channel, same severity scheme, no new design needed at that time beyond wiring. Do not design Kuma's specific check list against hardware that isn't racked.

**Design lock — no historical backfill, any domain.** The same principle from Decision 1 applies to every domain here, not just email: when a new source is wired into the relay — home automation, network, homelab once built, VPS/software, or any future addition — it starts monitoring and alerting from its activation point forward only. Onboarding a domain never means importing or processing its historical data through the new pipeline: wiring YoLink into the relay does not mean replaying weeks of past temperature readings; wiring UniFi does not mean backfilling the 5-week adoption-gap's historical logs into an alert stream; a future Kuma deployment does not mean reprocessing homelab history that predates it. This is a design lock, stated explicitly so nobody building against this ADR later assumes backfill is expected — or safe to add speculatively — for any source, present or future.

### The non-negotiable: heartbeat on the relay itself

Per B.J.'s single most important finding, this ships **in the same change** that generalizes the relay, not as later hardening: a two-layer pattern —
1. An independent Healthchecks.io dead-man's-switch pinged on a fixed schedule (proves the heartbeat mechanism itself ran).
2. A human-visible canary message sent *through* the relay on a fixed cadence, distinctly tagged (e.g., "relay heartbeat OK"), so Jeff would notice its *absence* the same way the fridge alarms' absence should have been noticed — a Telegram-token revocation fails exactly this way (silent), so the check must be something Jeff would miss seeing, not just a log line.

**Resolved — the canary runs as a Desktop Scheduled Task, same as everything else in v1.** An earlier draft of this ADR flagged a real tension here: a heartbeat's whole value is being independent of what it monitors, and a canary sharing the main runner's laptop-awake fragility is weaker evidence than an always-on VPS-based canary would be. **Jeff ruled on it directly — "desktop scheduled is fine to start with"** — accepting that tradeoff deliberately rather than by oversight: the canary proves "the laptop was awake and the relay's send-path worked," which still catches the actual failure mode this whole effort is about (a silently-dead bot token), just not "the laptop itself was off," which is a narrower, informed, and consciously accepted v1 gap — consistent with the same "down the road we have to change" framing that governs the rest of Decision 1. Revisit alongside the rest of Option 1 if/when v2+ externalization happens.

If per-channel bot tokens are used (four channels), the canary fires through all four channels on its schedule, not just one — a single-channel canary would not catch a different channel's token silently dying.

Ledger verifies this pairing actually catches a real outage once, shortly after it ships (a controlled test: rotate one channel's bot token deliberately, confirm the fidelity check surfaces it *for that channel specifically*, and confirm the other three channels are unaffected — proving both the detection and the blast-radius-containment claim above) — the same "should be working is not confirmation" standard WS-006/WS-007 already hold every other deploy to.

## Cross-cutting decision — human-in-the-loop confirmation, generalized

GL-018's propose-then-confirm rule is today written as an email-specific rule. It is generalized here into a system-wide invariant that applies to every domain in this ADR:

1. **No domain may auto-create a calendar event, task, ticket, or send an outbound message/command without Jeff's explicit approval.** This was already true for email; it now applies identically to a home-automation "should I create a task to fix the fridge" proposal, a network "should I file a ticket with the ISP" proposal, or anything else the relay's producers surface.
2. **The alerting layer's only autonomous action is *notify*.** It is not a remediation engine. Home Assistant *can* trigger automations that act on the physical world — this ADR draws the line that HA integrations in scope here are notify-only. Extending any domain from "propose" to "auto-remediate" (e.g., "if fridge >40°F for 1hr, auto-order a part") is a separate, explicit future decision requiring its own ADR and Jeff's approval — never an incremental scope-creep of this one.
3. **Reuse the existing task mechanism, don't invent a parallel one.** A home/homelab/network/VPS proposal that needs Jeff's nod and won't be resolved this turn should become a task via [[SOP-010-create-task]], exactly as SOP-023's `ACT` proposals already do — same backlog, same SSOT discipline, not a bespoke "home-automation proposal" format. One open question this surfaces, left to the implementer: does a proposal awaiting Jeff's confirmation need a distinct task `status` (e.g., `proposed`) separate from ordinary `open`, so "queued idea awaiting a yes/no" is visually distinct from "confirmed, ready to work"? That's a GL-004/task-schema call, not this ADR's to make.
4. **Destructive or infrastructure-changing actions stay gated exactly where they already are** — WS-007's irreversible-tier plan-and-approval gate for infrastructure, WS-006's design/build/deploy gates for software. This ADR's alerting layer does not create a new execution path around either; it only proposes into them.
5. **No domain backfills historical data through the alerting pipeline.** Every source — email included — starts monitoring from its activation/onboarding point forward only, never retroactively. This is a design lock, not an option weighed against alternatives; see the explicit statements under Decision 1's "Multi-inbox onboarding" and Decision 2's "Per-domain ingestion" sections above, which this point cross-references rather than restates.
6. **A confirmable proposal can resolve to more than one approved outcome, and the approval surface must present all of them.** Decision 1's draft-vs-send resolution is the first concrete instance of this: "approve" is not always a single binary. Wherever a proposal has more than one legitimate way to be approved (e.g., draft-only vs. send-now), the confirmation UI must expose the distinct outcomes explicitly rather than collapsing them into one generic "approve?" — this is a UI/prompt-design requirement, not a policy loosening of point 1 above.
